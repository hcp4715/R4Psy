3. 遇到复杂任务时，优先让模型帮你澄清需求、拆步骤、解释报错，而不是一次索要“完整答案”。

## 第 6 讲：数据预处理

### 任务 1：先澄清需求，再开始规划

#### 任务场景

当你知道自己要做“数据预处理”，但还没有把目标、变量、输出形式、限制条件说清楚时，不要直接让模型写代码。先让模型在 Planning 模式下持续追问，把关键前提问全。

#### Prompt 模板

```text
我正在做数据预处理任务。在写任何代码之前，请先在 Planning 模式下持续追问并澄清我的需求。
不要擅自做未说明的假设。请一直问到关键前提都被澄清。

请你重点澄清：
1. 我当前的数据是什么，包含哪些关键变量；
2. 我的目标结果是什么；
3. 中间需要哪些步骤；
4. 哪些约束会影响实现（如包、路径、缺失值、输出格式）；
5. 我还没有说明但必须确认的信息。

在你确认关键前提都已澄清之前，不要开始写代码。
```

#### 替换说明

- 把“数据预处理任务”替换成你当前在做的具体任务，例如“问卷数据清洗”“实验数据整理”。
- 如果你已经知道变量名，可以提前告诉模型，例如 `Site`、`age`、`weight_kg`。
- 如果你必须使用某个包，例如 `dplyr` 或 `data.table`，要提前写进 prompt。

#### 可继续追问

```text
请你根据我刚才补充的信息，把任务拆成最小可执行步骤，并说明每一步的预期输出。
```

```text
请你先不要写完整代码，而是先告诉我这个任务里最容易出错的 3 个地方是什么。
```

### 任务 2：先理顺从初始状态到目标状态的过程

#### 任务场景

当你还没有想清楚自己到底卡在“数据结构”“分析步骤”“变量定义”还是“实现限制”时，先让模型帮你把问题结构理顺，而不是直接求代码。

#### Prompt 模板

```text
我现在想解决一个问题，但我还没有完全想清楚思路。
请不要直接给我答案，而是先帮助我理顺从初始状态到目标状态的过程。

请按下面顺序引导我：
1. 帮我界定当前的初始状态：我现在已有的信息、数据、条件分别是什么？
2. 帮我界定目标状态：我最终想得到的结果是什么？怎样算完成？
3. 帮我识别中间路径：从现在到目标之间，通常需要拆成哪几个步骤？
4. 帮我找出缺失信息：哪些关键前提还不清楚，不能直接假设？
5. 请用简洁的结构总结：当前状态、目标状态、关键约束、可行步骤，以及我下一步最应该先确认的问题。

在关键前提没有澄清之前，不要直接进入代码或最终方案。
```

#### 替换说明

- 这一条通常不需要改很多，只要在第一句后补一句你当前在做什么即可。
- 如果你已经有部分代码或数据文件，最好提前告诉模型它们的名称与位置。
- 如果你不确定是“方法问题”还是“代码问题”，也可以直接写“我不知道自己卡在哪一步”。

#### 可继续追问

```text
请把你刚才总结的可行步骤改写成一个更适合初学者执行的版本，每一步都说明为什么要做。
```

```text
如果我现在只想先完成第一步，请告诉我第一步最少需要确认哪些信息。
```

### 任务 3：把小而具体的分析目标讲清楚

#### 任务场景

当你的任务已经比较明确，例如“筛选数据”“计算 BMI”“按组汇总”，最有效的做法不是把一堆要求混在一起，而是用统一结构把背景、数据、需求、约束交代清楚。

#### Prompt 模板

```text
背景：我正在学习第 6 讲的数据预处理。

数据：我已经加载了数据框 df，包含 Site、age、weight_kg、height_cm、sex。

需求：
1. 筛选出 Site 为 "Tsinghua" 且 age > 25 的被试；
2. 计算这些被试的 BMI；
3. 按 sex 输出 BMI 的均值和标准差。

约束：
- 请使用 dplyr；
- 正确处理缺失值；
- 每一步添加注释；
- 先解释思路，再给完整代码。
```

#### 替换说明

- `背景` 里写你正在做哪一讲、哪一道练习。
- `数据` 里写清楚数据框名称、变量名、数据类型或文件路径。
- `需求` 尽量写成 2–4 个明确步骤。
- `约束` 里写清楚包、输出格式、是否需要处理缺失值、是否要先解释思路。

#### 可继续追问

```text
请你检查我这个 prompt 里还有哪些地方说得不够具体，会影响你给出的代码质量。
```

```text
请你把这个任务拆成 3 个独立的小目标，并为每个小目标分别写一个更短的 prompt。
```

### 任务 4：用 `OpenCode` 把方案设计变成 coding

#### 任务场景

当任务有多个步骤、多个变量、多个文件时，不要立刻让 `OpenCode` 实现。先让它进入 Planning 模式，把需求问清楚，再确认方案，最后才执行。

#### Prompt 模板

```text
请先进入 Planning 模式，不要直接写代码。
先问我问题，直到关键前提都被澄清。
然后给出一个清晰的实现计划，说明每一步要做什么、预期输出是什么。
```

#### 替换说明

- 如果你在 `OpenCode` 中操作，可以把这条 prompt 放在最开始。
- 如果当前任务涉及多个文件、多个步骤、或者你还没看清楚数据结构，这条尤其适合。
- 看到计划后，不要直接默认它正确；先确认计划是否真的覆盖了你的任务。

#### 可继续追问

```text
请你把这个计划改写成更适合初学者理解的版本，并指出哪些步骤最值得我自己先动手做。
```

```text
我已经看过计划了。请你解释一下为什么这些步骤要按这个顺序执行。
```

## 第 7 讲：Tidyverse 数据预处理与实操演练

### 任务 1：批量导入与整理实验数据前，先澄清数据处理流程

#### 任务场景

第 7 讲的实验数据处理任务通常涉及多个 `.out` 文件、批量读取、筛选有效试次、按条件汇总、长宽转换和计算指标。这类任务不适合一句“帮我处理一下数据”。

#### Prompt 模板

```text
我正在做实验数据处理任务。在写任何代码之前，请先在 Planning 模式下持续追问并澄清我的需求。

请你重点帮我确认：
1. 数据文件放在哪里，文件格式是什么；
2. 需要保留哪些变量；
3. 无效试次的筛选标准是什么；
4. 需要按哪些条件进行 group_by 和 summarise；
5. 最终输出的数据结构应该是什么样子；
6. 哪一步最可能因为变量类型或列名不一致而出错。

在这些前提没有澄清之前，不要直接开始写代码。
```

#### 替换说明

- 把“实验数据处理任务”替换成你的具体任务，例如“计算 match-moral 条件下的 SPE”。
- 明确写出文件目录，例如 `slides/data/match/`。
- 提前告诉模型你打算用 `for loop`、`lapply`、`bind_rows`、`pivot_wider` 还是其他做法。

#### 可继续追问

```text
请你根据我们刚才澄清的内容，先只写出数据处理流程图或步骤说明，不要直接生成完整代码。
```

```text
请你指出这个实验数据任务里最容易出现的 3 类报错，并解释每一类通常是由什么原因造成的。
```

### 任务 2：把实验数据处理目标写成一个小而具体的 prompt

#### 任务场景

当你已经知道最终要算什么，例如 `SPE`、均值、准确率、宽表格式，就应该把分析任务拆成小而具体的目标，而不是把“读取文件 + 清洗 + 汇总 + 算指标”混在一句话里。

#### Prompt 模板

```text
背景：我正在学习第 7 讲的实验数据处理。

数据：我的数据文件位于 slides/data/match/，是多个 .out 文件。关键变量包括 Sub、Shape、Match、ACC、RT。

需求：
1. 批量读取所有被试数据；
2. 筛选有效试次（ACC 合法且 RT 在指定范围内）；
3. 按被试和条件计算平均 RT；
4. 在目标条件下进行长转宽；
5. 计算自我优势效应 SPE。

约束：
- 请使用 tidyverse；
- 每一步都解释为什么要做；
- 如果某一步依赖前一步输出，请明确写出来；
- 先给处理思路，再给代码。
```

#### 替换说明

- 如果你不是算 `RT`，而是算 `ACC` 或其他指标，记得在 `需求` 中改掉。
- 如果你的筛选标准不同，要在 `需求` 或 `约束` 中写明。
- 如果你想练习 `for loop` 或 `lapply` 的不同写法，也可以在 `约束` 中提前说明。

#### 可继续追问

```text
请你把这个任务拆成 5 个可以逐步验证的小任务，并说明每一步应该检查什么结果。
```

```text
请你先不要写最终版本代码，而是先写第 1 步和第 2 步，并告诉我怎样检查这两步是否正确。
```

### 任务 3：用 `OpenCode` 执行多步骤分析任务

#### 任务场景

第 7 讲的任务天然是多步骤的，所以适合先让 `OpenCode` 生成计划，再根据计划分步执行，而不是一上来直接 `/start-work`。

#### Prompt 模板

```text
请先进入 Planning 模式，不要直接写代码。
先问我问题，直到关键前提都被澄清。

然后请你生成一个分步骤的实现计划，说明：
1. 每一步要处理什么；
2. 每一步的预期输出是什么；
3. 哪些地方需要我人工确认；
4. 哪些步骤适合最后再统一检查。

等我确认计划后，再进入实现。
```

#### 替换说明

- 如果你已经准备好数据路径、变量名和目标指标，可以提前补充给模型。
- 如果你不确定计划是否合理，可以要求模型先解释计划，再决定是否继续。
- 在 `OpenCode` 里，如果计划已经生成，通常要先看计划内容再决定是否 `/start-work`。

#### 可继续追问

```text
请你根据这个计划，告诉我哪些步骤最适合我先手动理解，哪些步骤可以交给工具自动完成。
```

```text
请你把这个计划中的每一步改写成“我现在应该检查什么”的形式。
```

### 任务 4：先写 `analysis_notes.md`，再让 AI 读你的分析逻辑

#### 任务场景

当你的分析逻辑已经开始变复杂，例如涉及多个筛选标准、变量定义、计算顺序、长宽转换规则时，最省力的方式不是每次在对话框里重新解释，而是先把分析逻辑写进 `analysis_notes.md`。

#### Prompt 模板

```text
Please read analysis_notes.md first.
This file describes my analysis logic, key variables, and calculation steps.
Based on this file, help me continue the R analysis.

Before writing code, please tell me whether any part of the analysis logic is still ambiguous or inconsistent.
```

#### 替换说明

- `analysis_notes.md` 里至少要写清楚研究问题、数据来源、关键变量、计算逻辑。
- 可以把这个文件放在数据文件夹里，也可以放在项目根目录。
- 如果你更新了分析逻辑，记得同步更新 `analysis_notes.md`，否则模型会基于旧逻辑回答。

#### 可继续追问

```text
请你先根据 analysis_notes.md 用自己的话复述我的分析逻辑，确认你理解正确之后再写代码。
```

```text
请你指出 analysis_notes.md 里还有哪些地方写得不够清楚，可能导致后续代码实现出现偏差。
```

### 任务 5：在其他 prompt 中配合使用 `analysis_notes.md`

#### 任务场景

如果你的任务本身已经比较明确，但仍然依赖完整的分析背景，那么最有效的做法是：主体 prompt 只描述当前小任务，并提醒模型先读 `analysis_notes.md`。

#### Prompt 模板

```text
Please read analysis_notes.md first.

在此基础上，我当前只想完成一个小任务：
请帮我写出“按被试和条件计算平均 RT”这一步的 R 代码。

要求：
- 只完成这一步；
- 先解释输入和输出；
- 再给代码；
- 如果 analysis_notes.md 中有不明确的地方，请先指出。
```

#### 替换说明

- “当前只想完成一个小任务”后面要换成你当前这一步的具体目标。
- 如果你要模型只完成其中一步，一定要明确写“只完成这一步”。
- 如果模型开始越级做后面的步骤，你可以提醒它回到当前小目标。

#### 可继续追问

```text
请你继续下一步，但先说明这一步和上一部之间的数据结构是如何衔接的。
```

```text
请你告诉我，如果我把这一步单独拿出来检查，应该看到什么样的数据结果。
```

## 第 8 讲：描述性统计与数据可视化基础

### 任务 1：选择合适图形类型

#### 任务场景

第 8 讲最常见的问题不是“怎么写 `ggplot2` 代码”，而是“我到底该画什么图”。当你面对一个研究问题或练习题时，先让模型根据变量类型、分析目标和你想表达的信息，帮你选择合适图形。

#### Prompt 模板

```text
我正在学习第 8 讲的数据探索与可视化。

请你不要直接给我 `ggplot2` 代码，而是先帮我判断：
对于我的数据和分析目标，最合适的图形类型是什么。

我的信息如下：
- 数据框名称：df
- 变量 1：[变量名 + 类型]
- 变量 2：[变量名 + 类型]
- 如果有分组变量：[变量名 + 类型]
- 我想回答的问题是：[研究问题或练习目标]

请你按下面顺序回答：
1. 先判断我的变量结构属于哪一类（单变量、双变量、是否分组、连续/分类）；
2. 推荐 2–3 种合适图形；
3. 说明每种图形各自最适合展示什么；
4. 告诉我在我的场景下，最推荐哪一种，以及为什么；
5. 如果某种图形看起来常见但并不适合，也请明确指出原因。
```

#### 替换说明

- `[变量名 + 类型]` 要尽量写具体，例如 `RT（连续变量）`、`Shape（分类变量）`。
- 如果你关心的是“分布”，告诉模型你想看单变量分布；如果你关心的是“组间差异”，告诉模型你要比较不同组。
- 如果你已经心里有一个候选图形，例如 boxplot，也可以让模型判断“它是否真的是最合适的选择”。

#### 可继续追问

```text
如果我想重点突出组间差异，而不是单纯展示原始分布，你会更推荐哪种图形？为什么？
```

```text
请你把你推荐的图形按“最适合初学者完成练习”到“更适合正式汇报”排序，并解释排序理由。
```

### 任务 2：排查 `ggplot` / 数据探索代码报错

#### 任务场景

当你在第 8 讲写图形代码时报错，最有效的求助方式不是只说“我的代码不工作了”，而是把代码、报错信息、变量类型、你预期看到的图一起告诉模型。这样模型才能判断问题是出在 `aes()`、数据类型、分组、缺失值、因子顺序，还是管道中的前一步。

#### Prompt 模板

```text
我正在做第 8 讲的数据探索 / ggplot 练习，下面这段代码报错了。

当前代码：
[把你的最小可复现代码贴在这里]

报错信息：
[原样粘贴报错信息]

相关变量信息：
- 数据框名称：[数据框名]
- 变量 1：[变量名 + 类型]
- 变量 2：[变量名 + 类型]
- 分组变量（如果有）：[变量名 + 类型]

我原本想得到的结果：
[你预期看到的图形或分析结果]

请你按下面顺序帮助我：
1. 先判断最可能出错的是哪一层（数据、变量类型、映射、几何对象、分组、缺失值等）；
2. 用通俗的话解释这条报错通常意味着什么；
3. 指出我这段代码里最值得优先检查的地方；
4. 给出最小修改方案；
5. 再告诉我怎样验证修复后的结果是否合理。

请不要一开始就重写整段代码；先解释问题，再修复。
```

#### 替换说明

- 一定要贴“最小可复现代码”，不要把无关代码也塞进去。
- 报错信息要原样复制，尤其是报错里出现的变量名、函数名和提示语。
- 如果你不确定变量类型，可以先运行 `str(df)` 或 `glimpse(df)`，把结果摘要告诉模型。
- 如果你怀疑是因子顺序、缺失值或分组问题，也可以提前说明自己的猜测。

#### 可继续追问

```text
请你告诉我，这个报错更像是“数据本身的问题”还是“ggplot 语法的问题”，并分别给出检查方法。
```

```text
请你不要直接给最终图形代码，而是先给我一个从“最容易检查”到“最可能解决问题”的排查顺序。
```

### 任务 3：解读描述性统计结果

#### 任务场景

当你已经算出了均值、标准差、中位数、极值等结果，但还不知道该怎么用自然语言解释它们时，可以先让模型帮你组织表述。

#### Prompt 模板

```text
我正在学习第 8 讲的描述性统计。
下面是我得到的结果：[粘贴均值、标准差、最小值、最大值等]

请你用适合初学者的方式解释这些结果各自说明了什么，
并提醒我哪些地方不能过度解读。
```

#### 替换说明

- 贴结果时尽量带变量名。
- 如果是按组汇总结果，也请说明分组变量。

#### 可继续追问

```text
请你把这段解释改成更适合课堂汇报时口头表达的版本。
```

### 任务 4：生成 `ggplot2` 基础绘图代码

#### 任务场景

当你已经明确要画什么图，但还不熟悉 `ggplot2` 语法时，可以让模型帮你写一个基础版本。

#### Prompt 模板

```text
我已经确定要画 [图形类型]。
```
